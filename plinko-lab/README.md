# Plinko Lab (Provably-Fair)

A provably-fair, interactive Plinko game built with Next.js, Prisma, and Canvas.

## How to Run Locally

### Prerequisites
- Node.js (v18 or higher)
- npm or yarn

### Setup
1. Clone the repository and navigate into it.
2. Install dependencies:
   ```bash
   npm install
   ```
3. The environment variable for the local SQLite database is already set in the `.env` file generated by default: `DATABASE_URL="file:./dev.db"`.
4. Ensure the database is pushed to your local SQLite file:
   ```bash
   npx prisma db push
   npx prisma generate
   ```

### Running the App
Start the development server:
```bash
npm run dev
```
Open `http://localhost:3000` to interact with the game.
Open `http://localhost:3000/verify` to access the verifier page directly.

## Architecture

This application consists of a tightly integrated monolith spanning the frontend client, backend application server, and database operations.
- **Frontend**: Next.js 16 (App Router), React, standard 2D Canvas for physics faking/animation, and Tailwind CSS (v4) for styling. 
- **Backend / API**: Next.js Route handlers (`/api/rounds/*` and `/api/verify`) acting as controllers executing the fairness validation and Prisma DB queries. 
- **Database**: Prisma ORM with a SQLite provider.
- **Engine**: The core cryptographic protocol utilizes Node's `crypto` module (for SHA256) and a Mulberry32-inspired `xorshift32` generator seeded deterministically matching the exact assignment specifications.

### Project Boundaries
1. Game flow starts un-authenticated.
2. The client fetches a commit via `/api/rounds/commit` (Creates an un-revealed database `Round`).
3. The client calls `/api/rounds/:id/start` with a `clientSeed` and `dropColumn`, triggering the server to deterministically generate the grid path and layout, but hiding the serverSeed from the response.
4. The client simulates the rigid body path visually.
5. The client calls `/api/rounds/:id/reveal` which unlocks the `serverSeed`, enabling a mathematically-rigorous verification for fairness.

## Fairness Spec

1. A random 32-byte `serverSeed` and 8-byte `nonce` are generated per round.
2. The UI collects an active `clientSeed` alongside bet specifics.
3. Post round-reveal, a `combinedSeed` is computed exactly as: `SHA256(serverSeed + ":" + clientSeed + ":" + nonce)`.
4. An `xorshift32` PRNG is instantiated, adopting its initial state via `parseInt` from the first 8 characters (big-endian 4 bytes) of the `combinedSeed`.
5. The peg map enforces standard distribution rows, mapped recursively and clamped via the PRNG outputs mapped from `[0, 1)`.
6. Drop bias applies a `-0.01` up to `+0.01` proportional adjustment across decisions.
7. Validation happens against strict rounding constraints (`Math.round(leftBias * 1000000) / 1000000`) before yielding deterministic peg paths.
